スキル名: /verify-structure
説明: ファイルの移動・統合・分割・リストラクチャリング後に、内容ロスがないことをdiffで検証する
前提: 構造変更操作が完了していること。変更前の状態がgitまたはバックアップで保持されていること

手順:
1. 構造変更の種類を特定する:
   - リストラクチャリング（構造変更・フォーマット変更）
   - テンプレート化（PJ固有→汎用化）
   - バックアップ復帰
   - ファイル移動・統合・分割
2. 変更前の状態を取得する（git diff or バックアップファイル）
3. 種類に応じた検証を実行する:
   - リストラクチャリング: 削除された行が新構造のどこかに保存されているか確認。特にキーワード（固有名詞・数字・分類名）の脱落チェック
   - テンプレート化: 削除する具体例の一覧を作り、プレースホルダー＋記入例として残すべきものがないか検証
   - バックアップ復帰: 現在のファイルがバックアップのスーパーセットであることを確認
   - 移動・統合・分割: 元の全行がいずれかの移動先に存在することを確認。パス参照の更新も確認
4. 検証レポートを出力する:

```
## 構造変更検証レポート
- 操作: {何をしたか}
- 対象ファイル数: {N}
- 検証結果:
  - 完全一致: {N}件
  - 現在がスーパーセット（追記あり・削除なし）: {N}件
  - 内容ロスあり: {N}件 ← 0でなければ修正必須
    - {ファイル名}: {失われた内容の要約}
```

5. 内容ロスがあれば修正を提案する

## なぜこのスキルが必要か
構造変更時に「整理」として記述を短縮すると、固有名詞やキーワードが脱落し、検索による発見性が低下する。参照先の中身が無傷でも、インデックスの情報ロスは実質的な知見の喪失になる。
