# Collection Routes — 情報収集ルート

> このプロジェクトの情報ソースと収集方法を定義する。
> 新しいソースが増えたら追記する。

## 収集ルート一覧
| ソース | 収集方法 | 頻度 | 格納先 | 備考 |
|---|---|---|---|---|
| 定例MTG | TLDV文字起こし→Claude Code整形 | 週次 | minutes/ | 決定事項はlogs/meeting_decisions.mdに転記 |
| メール | 手動コピー or 転送 | 都度 | minutes/ or logs/ | 合意事項は信頼度「確定合意」で記録 |
| Slack | 手動コピー or API自動取得 | 都度 | **data/slack_raw.md** に原文追記 → logs/に転記 | 揮発性ソース。原文蓄積必須 |
| 共有ドキュメント | URL参照 or スナップショット保存 | 変更時 | refs/ | 版管理に注意 |
| {追加ソース} | | | | |

## 情報フロー
```
[1次ソース] → minutes/（一次着地点）
                ↓ 抽出・分類
         ┌──────┴──────┐
    logs/（追記型）  reports/（加工済み）
    - 意思決定       - 週次報告
    - スコープ変更    - 状況分析
    - 予算変動       - ステークホルダー向け資料
    - 技術選定
         │
    refs/（詳細資料）← ログからリンク
```

## 収集時の注意
- 1次情報はまずminutes/に格納。分類・抽出はその後
- ソースの信頼度をエントリ記録時に必ず付与する（WRITING_RULES.md参照）
- 口頭情報は可能な限り早くテキスト化する（記憶の劣化を防ぐ）

### Slack原文の保全ルール
- **Slackは揮発性ソース**: 流れてしまうと再取得が困難。会話で共有されたSlackの内容は即座に `data/slack_raw.md` に追記する
- **ログ転記とは別に原文を蓄積**: ログファイルへの転記（要約・構造化）とは独立して、原文をそのまま保存する
- **フォーマット**: 日付 / チャンネルor相手 / 発言者 / 原文。構造化・要約はしない
- **自動収集への移行**: 将来Slack APIによる自動取得を導入する場合、同じファイル（`data/slack_raw.md`）に同じフォーマットで追記する。手動→自動の切り替え時にファイル構造の変更は不要
