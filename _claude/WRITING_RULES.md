# Writing Rules — 記録品質ルール

## 原則: 再現可能性
記録から「同じ判断を再現できる」ことが最優先。
抽象化して情報を落とすくらいなら、具体のまま残す。

## 3つの記録型
- **追記型**（時系列エントリを追加、上書きしない）: logs/のファイル。意思決定・合意の履歴保全。各エントリに日付/ソース/信頼度/内容を記録
- **スナップショット型**（日付付き新規ファイル）: 週次報告、状況更新、ステークホルダー向け資料。時点の記録が重要なもの
- **更新型**（1ファイルを更新し続ける、常に最新が正）: 合意事項トラッカー、現行スコープ定義、体制表。「今どうなっているか」が1箇所でわかる必要があるもの
- 変更履歴はgitで追跡する

## インデックスによるコンテクスト量の制御
- 各フォルダのREADME.mdにインデックス（ファイル一覧＋1行要点）を維持する
- Claude Codeはまずインデックスを読み、必要なファイルだけアクセスする。全ファイルを読まない
- インデックスの更新はファイル追加時にセットで行う

## エントリフォーマット
すべてのログエントリに以下を必須とする:

| 項目 | 説明 |
|---|---|
| 日付 | YYYY-MM-DD |
| ソース | 情報の出所（MTG名、メール、Slack、資料名など） |
| 信頼度 | 下記5段階のいずれか |
| 内容 | 事実と判断を分離して記述 |

## 信頼度分類（5段階）
| レベル | 定義 | 例 |
|---|---|---|
| 確定合意 | 書面（契約書、メール）で合意済み | 契約書に記載、メールで承認 |
| 口頭合意 | MTG等で口頭合意。議事録に記載 | 定例MTGで合意、電話で確認 |
| こちらの認識 | 自分の解釈。先方の明示的合意なし | 仕様書からの読み取り、過去経緯からの推定 |
| 先方主張 | 先方が主張しているが、こちらは合意していない | 先方メールでの要求、MTGでの発言 |
| 推測 | 状況からの推測。要確認 | 類似案件からの類推、市場動向からの推定 |

## 記述のアンチパターン

### NG: 判断ロジックの省略
- NG: 「Aを採用した」
- OK: 「A/B/Cを比較し、コスト（A:50万, B:80万, C:120万）と納期（A:2週, B:3週, C:1週）のバランスからAを採用。Cは納期最短だがコスト超過」

### NG: 数字の丸め・省略
- NG: 「コスト増」「予算オーバー」
- OK: 「+150,000円（当初見積300,000円→実績450,000円）」

### NG: 主語の省略
- NG: 「承認された」
- OK: 「田中部長が承認（2025-03-01 定例MTGにて口頭）」

### NG: 過度な抽象化
- NG: 「スコープを調整した」
- OK: 「機能Xのうち、サブ機能X-2, X-3を次フェーズに延期。理由: 納期固定のため優先度A（X-1, X-4）に集中」

## 事実と判断の分離
- 事実: 「先方から○○と言われた」「データ上○○である」
- 判断: 「したがって○○すべき」「○○のリスクがある」
- 記録時は事実→判断の順で書き、判断には根拠を付ける

## 構造変更時の検証ルール

ファイルの移動・統合・分割・リストラクチャリング・テンプレート化・バックアップ復帰など、**既存コンテンツの構造を変える操作**を行った場合、以下の検証を必ず実行する。

### なぜ必要か
構造変更時に「整理」として記述を短縮すると、検索キーワードとして機能する固有名詞（分類名・シナリオ名・数値等）が脱落する。参照先の中身が無傷でも、インデックスの発見性が低下し、実質的な情報ロスになる。

### 対象操作と検証手順

#### 1. リストラクチャリング（構造変更・フォーマット変更）
- 変更前のファイルを保持（バックアップまたはgit commit）
- 変更後に `diff` で旧→新を比較
- **削除された行が1行でもあれば、その行の情報が新構造のどこかに保存されているか確認**
- 特にインデックス・目次・Key Files記述：キーワード（固有名詞・数字・分類名）は削除禁止。追加のみ

#### 2. テンプレート化（PJ固有→汎用化）
- 汎用化で削除する具体例・固有名詞を一覧化する
- 削除する各項目について：「この具体性がなくても、テンプレート利用者が何をすべきか判断できるか？」を検証
- 判断できない → プレースホルダー＋記入例として残す（例: `{チーム名} — 例: Kteku`）
- PJ固有の知見は別ファイルに退避してからテンプレート化する。消さない

#### 3. バックアップ復帰
- バックアップ内の全ファイルを一覧化
- 各ファイルの復帰先パスを明示的にマッピング（テーブルで）
- 復帰後に全ファイルを `diff` 比較
- **現在のファイルがバックアップのスーパーセットであることを確認**（バックアップにあって現在にない行がゼロ）
- マッピング先がない（新規ファイル）場合は、そのファイルの内容がどこかに吸収されたか確認

#### 4. ファイル移動・統合・分割
- 移動元の全行が移動先に存在することを `diff` で確認
- 統合時：統合元の各ファイルの全行が統合先に含まれることを確認
- 分割時：分割前の全行がいずれかの分割先に含まれることを確認
- パス参照（`→ filename.md`、`refs/xxx.md` 等）が全て新パスに更新されていることを確認

### 検証の出力フォーマット
構造変更後に以下を出力し、ユーザーに確認を求める：

```
## 構造変更検証レポート
- 操作: {何をしたか}
- 対象ファイル数: {N}
- 検証結果:
  - 完全一致: {N}件
  - 現在がスーパーセット（追記あり・削除なし）: {N}件
  - 内容ロスあり: {N}件 ← 0でなければ修正必須
    - {ファイル名}: {失われた内容の要約}
```
