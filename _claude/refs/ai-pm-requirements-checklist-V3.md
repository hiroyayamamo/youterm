# AIプロジェクト管理・要件定義 全量チェックリスト（v3: クライシス対応強化版）

> AIプロジェクトの全フェーズで見落としがちな観点を網羅的に整理したもの。
> 各項目は質問形式で、プロジェクト開始時のヒアリングやセルフチェックに使用する。
> カテゴリは**領域別**に分類し、フェーズ間の重複を排除している。
>
> **v3改善方針**: v2（9カテゴリ・約140項目）に対し、実案件のクライシス事例から得た教訓を反映。
> - 契約・ガバナンス領域の防御的観点を大幅強化
> - 「緩く入って徐々に締める」契約ライフサイクル戦略を追加
> - PM価値の可視化・エビデンス保全の観点を追加
> - 参照: NIST AI RMF / EU AI Act / 経産省AIガバナンスGL / ISO 42001 / PMBOK / OWASP Top 10 for LLM

## 0. プロジェクト計画（スコープと導入アプローチ）

### 0-1. スコープ定義
- [ ] プロジェクト全体のゴール（何を実現し、何を実現しないか）は明文化されているか？
- [ ] ターゲットユーザーのペルソナ、利用コンテキスト（誰が・いつ・どこで・何のために使うか）、主要ユースケースは具体的に定義されているか？
- [ ] AI活用が必要な領域と、従来手法で対応する領域の切り分けは合意されているか？
- [ ] 成果物の定義（納品物一覧、完了条件）は明確か？
- [ ] **成果物に含まれないもの（除外リスト）は明示されているか？** 後から「出していない＝不履行」と主張されるリスクを防ぐ

### 0-2. スコープと契約・予算の整合
- [ ] スコープに対して予算は妥当か？予算制約からスコープを逆算する検討はされているか？
- [ ] 納品可能な成果物（システム、モデル等）は請負契約、不確実性の高い探索（PoC等）は準委任契約と、スコープの性質に応じた契約形態の使い分けは検討されているか？
- [ ] 個人の人月単価ではなくチーム単位での受注など、利益率を確保できる体制設計になっているか？
- [ ] 見積もり時点で不確実性が高い領域は、概算見積もり＋再見積もりのタイミングを設けているか？
- [ ] **契約ライフサイクル戦略（緩く入って徐々に締める）は意図的に設計されているか？** 初期の曖昧さを放置すると、後に攻撃材料になる

### 0-3. スコープ変更管理
- [ ] スコープ変更が発生した場合の変更管理プロセス（申請→影響評価→承認→反映）は合意されているか？
- [ ] 納品型プロジェクトの場合、スコープクリープ（範囲の際限ない拡大）を防ぐ仕組み（変更管理委員会、追加見積もり等）はあるか？
- [ ] 伴走型・コンサルティング型プロジェクトの場合、スコープ変更を前提とした範囲設計（期間・工数ベースの契約、定期見直し等）になっているか？
- [ ] 「やらないこと」の明示と、ステークホルダーとの合意は文書化されているか？
- [ ] **仕様変更は書面（メール・チャット含む）での依頼→影響評価→書面合意→着手のプロセスが確立されているか？** 口頭での「ちょっと変えて」を防ぐ
- [ ] **仕様変更に伴う追加コスト・スケジュール影響は、変更発生時点で即座に先方に伝達・合意されているか？** 異議なく対応すると「合意の上」と解釈される

### 0-4. 仕様・成果ドキュメント管理
- [ ] 最終提出する成果ドキュメントの一覧と、中途段階での保存・管理方法（バージョン管理、保管場所等）は決まっているか？
- [ ] 仕様の合意→更新→変更→オミット（取り下げ）の各イベントをドキュメントに反映し、常に最新状態を保つ運用ルールは確立されているか？
- [ ] 仕様変更時のロールバック（過去バージョンへの復元）が必要になった場合の手順は想定されているか？
- [ ] 「誰が・いつ・何を・なぜ変更したか」の変更履歴を追跡できる仕組み（Git、変更履歴シート等）はあるか？
- [ ] ドキュメントの正本（Single Source of Truth）はどこに置くかが全員に周知されているか？
- [ ] **「ドキュメントを作成しない」という判断（先方指示含む）も記録されているか？** 後から「出していない＝管理不全」と攻撃される防御
- [ ] **フェーズごとの定型ドキュメント一覧をキックオフ時にクライアントと合意しているか？**「各フェーズでこういうものを出します」を最初に伝えておく。後出しにしない（refs/pm-document-guideline.md参照）
- [ ] **納品物一覧を契約書に明記し、含まれないものを除外リストに記載しているか？** 後から「これも含まれていたはず」を防ぐ
- [ ] **先方が不要と言っても定型ドキュメントは最低限作成・共有しているか？** 省略する場合でも省略の合意を書面で記録する

### 0-5. 段階的導入（フェーズゲート）
- [ ] PoC（概念実証）のスコープ・期間・成功基準は定義されているか？
- [ ] PoC結果をもとに本開発に進むかのGo/No-Go判断基準は合意されているか？
- [ ] パイロット導入の対象範囲（特定部門、特定ユースケース等）は決まっているか？
- [ ] パイロットから全社展開へのスケールアップ計画はあるか？
- [ ] 各フェーズの移行基準（次フェーズに進むための条件）は明確か？
- [ ] **クライアントがOKを出していても、開発期間中に業務理解が進んだり要件が変更される可能性への対策はスケジュールに組み込まれているか？** 受注側に法的責任がなくても「使えない」で継続発注が止まるリスクがある。対策例: 1ヶ月程度の受入修正期間の設定、小規模PoCでの業務理解度の吸収。少なくとも開発着手前にこのリスクを指摘し、対策の要否をクライアントと協議しておくことで信頼を担保できる

### 0-6. スケジュール・マイルストーン
- [ ] フェーズ別マイルストーン（PoC完了、MVP完了、GA等）は設定されているか？
- [ ] AI特有の不確実性（精度が読めない等）を加味したスケジュール設計になっているか？
- [ ] チケット単位での進捗追跡の仕組みは構築されているか？
- [ ] **マイルストーン単位の評価サイクル（評価→継続/変更/中止の判断ポイント）は合意されているか？** 仕様凍結とスコープ見直しのタイミングを制度化する

### 0-7. 契約ライフサイクル戦略（Loose-to-Tight）

> 契約の曖昧さは初期の受注を容易にするが、放置すると後に攻撃材料になる。
> 意図的に「緩く入って徐々に締める」プロセスを設計する。

- [ ] **Phase 1（獲得期）**: 初期契約の曖昧さは意図的なものとして認識されているか？ 放置すべきでない曖昧ポイントのリストは作成されているか？
- [ ] **Phase 2（硬化期）**: プロジェクト進行に伴い、小さな覚書（Amendment）で順次明文化するスケジュールは設定されているか？
- [ ] **Phase 3（成熟期）**: プロジェクト中盤までに、全ての「付随業務」が契約・経済ルールに紐づいているか？
- [ ] **危険信号**: 初期の曖昧さが中盤以降も残っている場合、それ自体をリスクとして認識・エスカレーションしているか？

## 1. ガバナンス・コンプライアンス（意思決定とリスク）

### 1-1. ビジネスゴールとROI
- [ ] プロジェクトの成功指標（KPI）は定量的に定義されているか？
- [ ] AI導入によるビジネス価値は、コストに見合うと判断できる根拠があるか？
- [ ] KPIの測定方法と測定タイミングは合意されているか？
- [ ] AI導入前の業務ベースライン指標（処理時間、コスト、品質等）は測定されているか？

### 1-2. 撤退判断
- [ ] 精度不足・赤字拡大時に中止する基準は事前に合意されているか？
- [ ] サンクコストにとらわれず判断するための意思決定プロセスはあるか？
- [ ] 撤退時のデータ・成果物の扱い（引き上げ・破棄）は決まっているか？
- [ ] **撤退時のデポジット精算・既履行分の算定方法は契約上明確か？**
- [ ] **撤退が経済的に成立するか（最大損失額）のシミュレーションは事前に実施されているか？**

### 1-3. 責任分解（RACI）とステークホルダー管理
- [ ] インシデント発生時の実行者・最終責任者は明確か？
- [ ] AI起因の損害が発生した場合の責任範囲は契約上整理されているか？
- [ ] 意思決定権限と承認フローは文書化されているか？
- [ ] 全ステークホルダー（経営層、現場部門、IT、法務、外部ベンダー等）は特定・一覧化されているか？
- [ ] ステークホルダー別のコミュニケーション計画（方法・頻度・内容）は策定されているか？
- [ ] **指揮命令権（業務遂行の方法・優先順位の決定権）が受託者側にあることは契約上明記されているか？** 準委任契約の場合は必須
- [ ] **体制変更は双方の書面合意を必須とする条項はあるか？** 一方的な体制変更は契約違反とする
- [ ] **コミュニケーション窓口は限定されているか？** 発注者から開発者への直接指示を禁止する規定

### 1-4. コンプライアンス・法規制
- [ ] 業界特有の法規制（薬機法、金商法、景表法等）への適合は確認済みか？
- [ ] AI関連法規制（EU AI Act等）に基づくリスク分類は実施されているか？
- [ ] 高リスクAIに該当する場合の適合性評価手順は整備されているか？
- [ ] 専門用語や表記の正当性を誰がレビューするか決まっているか？
- [ ] **準委任契約で先方が指揮命令権を持つ体制は偽装請負リスクがあることを認識しているか？**

### 1-5. 知的財産権の帰属
- [ ] AI生成物の著作権・所有権の帰属先は契約上明確か？
- [ ] 独自開発したプロンプトの知的財産権は整理されているか？
- [ ] 学習データに第三者の著作物が含まれる場合のリスク評価はされているか？
- [ ] **ソースコードの知的財産権の帰属（発注者 or 受託者）は明確か？** 帰属先によって交渉ポジションが変わる

### 1-6. AI倫理・公平性
- [ ] 生成結果の公平性（性別・人種・年齢等のバイアス）に関する方針はあるか？
- [ ] 差別的・有害な出力を防止する仕組み（ガードレール）は設計に含まれているか？
- [ ] ユーザーに対してAI利用であることの開示方針（ラベリング・透かし含む）は決まっているか？
- [ ] AIの意図しない用途（悪用・誤用）のシナリオは洗い出され、対策されているか？

### 1-7. AIリスク管理
- [ ] AIリスクアセスメントの手法（リスクマトリクス等）は定義されているか？
- [ ] 識別されたリスクの優先順位付けと対応計画（受容/軽減/移転/回避）は策定されているか？
- [ ] AIシステムが社会・環境に与える影響（雇用、格差、CO2排出等）は評価されているか？

### 1-8. Human Oversight（人間による監視）
- [ ] AIの判断に対する人間のレビュー・介入・オーバーライド機能は設計されているか？
- [ ] 人間の監視方式（HITL / HOTL / HITC）と自動化レベルの段階はユースケースに応じて選定されているか？

### 1-9. 組織AIガバナンス
- [ ] 組織としてのAI利用方針（AIポリシー）は策定・公開されているか？
- [ ] AI倫理委員会やAI CoE等の横断的ガバナンス体制は設置されているか？
- [ ] 経営層がAIガバナンスにコミットする仕組み（定期レビュー等）はあるか？

## 2. 組織・人材（体制とケイパビリティ）

### 2-1. チェンジマネジメント
- [ ] AI導入により変更される業務プロセスは特定・文書化（As-Is / To-Be）されているか？
- [ ] 影響を受ける部門・担当者への事前説明と合意形成は計画されているか？
- [ ] AI導入に対する組織内の抵抗（心理的障壁）への対応策は検討されているか？
- [ ] 既存業務からAI活用業務への移行期間・移行方法は計画されているか？

### 2-2. 教育・トレーニング
- [ ] エンドユーザー向けの利用マニュアル・トレーニング計画は策定されているか？
- [ ] 運用担当者向けの技術トレーニング（プロンプト調整、障害対応等）は計画されているか？
- [ ] AI活用のベストプラクティスやガイドラインは整備されているか？
- [ ] ナレッジの属人化を防ぐためのドキュメント整備・引き継ぎ計画はあるか？

### 2-3. チーム体制・人材確保
- [ ] AI開発に必要な役割（MLエンジニア、プロンプトエンジニア、ドメインエキスパート等）は定義・アサインされているか？
- [ ] 必要スキルに対する現有メンバーのスキルギャップは評価されているか？
- [ ] コア技術の内製 vs 外注の判断基準は明確か？
- [ ] キーパーソン離脱時のリスク対策（バックアップ要員、ドキュメント等）はあるか？

## 3. 契約・コスト管理（調達とリソース）

### 3-1. 契約・SLA
- [ ] AI関連の契約形態（準委任/請負/SaaS等）はAI開発の不確実性を考慮して選定されているか？
- [ ] AIサービスの可用性SLA（稼働率、レスポンスタイム等）は定義・合意されているか？
- [ ] SLA未達時のペナルティ条項は契約に含まれているか？
- [ ] 契約更新・解約条件（通知期間、データ引き渡し等）は明確か？
- [ ] 瑕疵担保（契約不適合責任）の範囲はAI特有の不確定性を考慮して定義されているか？
- [ ] **個別契約（個別発注書）は案件ごとに締結されているか？** 見積もり了承のみでは法的拘束力が弱い
- [ ] **見積もり了承の記録（メール・チャット・録画等）はエビデンスとして保全されているか？**
- [ ] **中途解約時の精算条項（既履行分の報酬請求権、違約金等）は明記されているか？** 民法648条3項の適用を確認
- [ ] **発注者都合の解約の場合、既履行分の全額精算＋残存期間の一定割合を違約金とする条項はあるか？**
- [ ] **債務不履行の定義は具体的か？** 曖昧な定義は「ドキュメント未提出＝不履行」等の拡大解釈に使われる

### 3-2. コスト管理
- [ ] 初期費用・ランニングコスト（API、サーバー、人件費）のシミュレーションは実施済みか？
- [ ] ピーク時と通常時のコスト変動は見積もられているか？
- [ ] 予算超過を防ぐアラート・上限設定は設計に含まれているか？
- [ ] コスト増加時のスケール判断基準は合意されているか？
- [ ] 運用フェーズの予算は開発フェーズとは別に確保されているか？
- [ ] **費用報告の粒度と頻度は合意されているか？** 「全発注の事前承認」は準委任の裁量と矛盾する
- [ ] **マージン構造（PM費用・管理費用の内訳）は社内で整理されており、問われた場合の回答が準備されているか？**

### 3-3. ベンダー管理
- [ ] AIベンダー・ツールの選定基準と評価プロセスは定義されているか？
- [ ] 特定LLMベンダーへの依存度は評価されているか？
- [ ] ベンダー乗り換え時のコスト・工数は見積もられているか？
- [ ] マルチモデル戦略（タスク別モデル使い分け等）の必要性は検討されているか？
- [ ] サードパーティAIコンポーネントの信頼性評価（デューデリジェンス）は実施されているか？

### 3-4. モデル更新・変更管理
- [ ] 利用中モデルが廃止（DEPRECATED）された場合の切り替え手順は想定されているか？
- [ ] モデル変更時の影響範囲を最小化する設計（抽象化レイヤー等）になっているか？
- [ ] 新モデル登場時の評価・移行プロセスは定義されているか？
- [ ] モデル/プロンプト変更が悪化した場合の切り戻し手順と判断基準は合意されているか？
- [ ] Blue-Greenデプロイやカナリアリリース等の段階的リリース戦略は検討されているか？

### 3-5. デポジット・前受金管理
- [ ] **デポジットの精算方法（稼働時間×単価 or マイルストーン単位）は契約上定義されているか？**
- [ ] **デポジット残高の定期報告と消化状況の共有ルールは合意されているか？**
- [ ] **中途解約時のデポジット返還条件（既履行分の算定基準、返還対象の定義）は明確か？**
- [ ] **既履行分のエビデンス（稼働記録、コミットログ、リリース実績、MTG議事録）は常時保全されているか？**

### 3-6. 再委託先（開発パートナー）管理
- [ ] **再委託先との契約に、発注者との直接取引禁止条項は含まれているか？** 契約期間中および終了後一定期間
- [ ] **再委託先との契約に、発注者からの直接の勧誘・引き抜きを禁止する条項はあるか？**
- [ ] **発注者と再委託先の間に直接のコミュニケーションが発生した場合の取り扱いルールは定められているか？**
- [ ] **再委託先が発注者から直接アプローチを受けた場合の報告義務は合意されているか？**
- [ ] **再委託先への発注済み案件の中途キャンセル条件（通知期間、違約金等）は把握されているか？**

## 4. データ管理・プライバシー（信頼性と安全）

### 4-1. 個人情報保護
- [ ] 入力・生成過程での個人情報の取り扱いルールは定義されているか？
- [ ] 必要に応じたマスキング・匿名化の要件は明確か？
- [ ] Pマーク・GDPR等、適用される法規制は特定されているか？

### 4-2. データ保管・リージョン管理
- [ ] データの物理的な保管場所（国内限定等）は指定されているか？
- [ ] クラウドベンダーのリージョン設定は法規制・セキュリティ要件を満たしているか？
- [ ] データの越境移転に関するリスク評価は実施済みか？

### 4-3. データライフサイクル
- [ ] 契約終了時のデータ削除手順と証明方法は合意されているか？
- [ ] 一定期間経過後の自動削除ポリシーは定義されているか？
- [ ] ログやキャッシュに残る派生データの削除範囲は明確か？
- [ ] ログ・監査データの長期保存要件（保存期間、ストレージ種別）は合意されているか？

### 4-4. データ品質・ガバナンス
- [ ] 学習・RAGに用いるデータのノイズ除去手順は確立されているか？
- [ ] 機密情報が学習データに混入しないチェック体制はあるか？
- [ ] データ品質の評価基準（正確性、完全性、代表性等）は定義されているか？
- [ ] データセットのバイアス評価は体系的に実施されているか？
- [ ] データの来歴（Data Lineage / Provenance）を追跡する仕組みはあるか？

## 10. AIエージェントの業務適用戦略（SoA設計）

> AIエージェントを業務に組み込む仕組みは、従来のSoR（System of Record: データの記録・保存・検索）とは根本的に異なる。
> 解釈し、判断し、行動するSystem of Action（SoA）には、SoA固有の導入・設計アプローチが必要となる。
> SoRの設計手法（トップダウン全体設計）をSoAにそのまま持ち込むと失敗する。
>
> **参照**: gura「AIエージェントで業務を丸ごと変えようとして、半年間失敗した」(2026-03)

### 10-1. SoR / SoA の区分とアプローチ選択
- [ ] AI活用領域がSoR（データの記録・保存・検索）かSoA（解釈・判断・行動）かを区別しているか？
- [ ] SoR領域にはトップダウン全体設計、SoA領域にはボトムアップ構築と、性質に応じたアプローチを選択しているか？
- [ ] 以下のようなトップダウン構想を、SoA領域に適用していないか？ いずれも半年間成果ゼロに終わった実例:
  - 「複数案件を横断して情報管理し、ステータス追跡、提案書生成から顧客フォローまで一気通貫で回す」（セールス全体の自動化）
  - 「キックオフを起点に何週間前に何をやる、何週間後に何をやるというタイムラインの自動化。PMトップが全PJの状態を把握できるよう、SlackやNotionから情報を集約して一元管理」（全PJ共通PM基盤）
  - → 結果: 全PJ共通の仕組みを作ったら、どのPJの問題すら1つも解けない仕組みが出来上がった
- [ ] SoAでは、エージェントが必要とする判断基準・ドメイン知識・暗黙のルールは事前に定義できず、実際に業務を回しながらでしか見えてこないことを前提としているか？
- [ ] トップダウンが失敗する構造的理由を理解しているか？ — 案件ごとにステークホルダー・業務委託・アサイン状況・売上見込みが全て異なり、情報のルーティング設計が破綻する。さらに「全体を集めてから分割」は循環依存に陥る（分割ロジック自体が全体コンテキストに依存する）

### 10-2. コンテキスト空間の設計と蓄積

> コンテキストの蓄積には2つの効き方がある:
> - **パイプライン化** = 同じ作業を毎回速くする（定常業務の高速化）
> - **コンテキスト蓄積** = 想定していなかった作業まで速くする（突発・非定常の対応力）
>
> コンテキスト × スキルは複利で効くアセット。業務を言語化してmdに残すこと自体が、将来の自分とAIエージェントへの投資になる。
> 早い段階で始めるほど、あとから想定外の場面で回収できる。

- [ ] エージェントが業務遂行に必要なコンテキスト空間（ステークホルダー、ドメイン用語、機密/共有の区分、報告フォーマット等の文脈の集合）は定義されているか？
- [ ] コンテキスト空間はCLAUDE.md等の設定ファイルに収まるサイズに制限されているか？ 小さいほど精度が高く速い
- [ ] 「全体のコンテキストを集めてから動的に分割する」アプローチを取っていないか？ 分割ロジック自体が全体コンテキストに依存する循環構造に陥る
- [ ] 散らばった情報を1箇所（プロジェクトフォルダ等）にMarkdown等で集約しているか？ 情報が集約されていればエージェントは力を発揮する
- [ ] コンテキスト空間はプロジェクト固有のステークホルダー・ドメイン知識・報告フォーマットに特化しているか？ 汎用化すると精度が落ちる
- [ ] 以下のタイミングで、業務の背景をAIに言語化させてmdに蓄積しているか？
  - **スキル作成時**: なぜこのスキルが必要か、どの業務課題を解決するか
  - **MTG終了時**: 意思決定の経緯、ステークホルダーの関心事・懸念、合意事項の背景
  - **ツール・技術選定時**: なぜこの選択をしたか、比較検討した代替案、選定理由
  - **ステークホルダーとのやり取り後**: 過去の経緯、相手の立場・関心事、やり取りの文脈
- [ ] 蓄積されたコンテキストが、定常業務のパイプライン化だけでなく、突発的な対応にもエージェントが自律的に対応できる基盤になっているか？ 具体例: ステークホルダーからプロジェクトで利用中のツールについて問い合わせメールが来た → AIがメールを読み込み → ツールがPJでどう使われているかをmdから把握 → 公式サイトで最新仕様を取得 → 「なぜこの質問をしているか」（ステークホルダーの関心事・懸念）もmdから読み取り → 相手の目線に立った説明文を作成。人間がやったのはメールの存在をAIに伝えただけ

### 10-3. ボトムアップ構築プロセス
- [ ] まず手動でAIに業務を依頼し、間違えたら指摘し、正しくできたら覚えさせるプロセスを経ているか？ 具体例: gog-cli経由でGmailのスレッドを取得させたら、古い順に全文を読み始めて最新のやり取りを見落とした → 「スレッドは末尾から読め」をスキルに明記。こうした試行錯誤の1つ1つが後のスキル設計の土台になる
- [ ] 2-3回同じことを頼んだタイミングでスキル化しているか？ いきなりスキル化しない。最初からパイプラインを設計しない
- [ ] 導入スコープは「1つのプロジェクト・1つの案件・1つの業務」に絞られているか？ 1つのPJなら、ステークホルダーは数人、ドメイン知識は1分野、報告フォーマットは1種類。CLAUDE.mdに書けるサイズに収まる
- [ ] 例外条件、他メンバーとの連携フロー、共通化は意図的に後回しにしているか？
- [ ] 「自分の手元の、この案件の、この業務だけ」に集中できているか？

### 10-4. 拡張と共通化の順序
- [ ] 拡張パスは「タスク単位 → 業務単位 → 責任範囲全体」の順序で設計されているか？
- [ ] 既存のコンテキスト空間を「拡張」するのではなく、新しい小さな空間を作って後から繋げるアプローチを取っているか？
- [ ] 共通化は「個別に作ったスキルがたまたま同じだった」場合にのみ行い、最初から共通基盤を設計していないか？ 実例: 音声文字起こしスキルをPM用・セールス用に個別に作ったら、たまたま同じものだった。それを共通化しただけ
- [ ] 「全案件共通の仕組み」「横断基盤の設計」「業務全体の自動化」を早すぎた最適化として認識しているか？

### 10-5. 情報アーキテクチャ（プロジェクトフォルダ設計）

> PM業務の本質は「情報の流れを設計すること」。
> プロジェクトフォルダが全情報のハブとなり、「ここを見ればわかる」状態を維持する。
>
> **参照フォルダ構成（gura氏の実例）**:
> ```
> project-root/
> ├── CLAUDE.md          — プロジェクトの脳。ステークホルダー、ドメイン知識、検索クエリテンプレート
> ├── .claude/skills/    — PM業務のビルディングブロック（スキル定義群）
> ├── minutes/           — 情報の一次着地点。音声データ、文字起こし、整理済み議事録
> ├── reports/           — 加工済みアウトプット。状況報告書、週間ワークフロー定義
> ├── docs/              — 業務棚卸し。定期的にAIにまとめさせた業務整理結果
> ├── work-description/  — タスク定義書。各業務タスクの手順と目的を1タスク1ファイルで明文化
> └── data/              — 分析用の生データ。CSV等、SQL集計の対象
> ```

- [ ] あらゆる情報が最終的に経由する「ハブ」となるプロジェクトフォルダが存在するか？
- [ ] フォルダは情報の性質別に役割が分かれているか？ 最低限以下の区分を持つ:
  - **一次情報の着地点**（音声、文字起こし、議事録等）— 同じフォルダ内で情報が段階的に加工される: 音声ファイルを置く → 文字起こしされる → 構造化された議事録になる
  - **加工済みアウトプット**（状況報告書、ステークホルダー向け資料等）
  - **業務定義**（タスクの手順・目的の明文化、業務棚卸し結果）
  - **生データ**（CSV等の分析対象データ）
- [ ] 情報の加工段階ごとにスナップショット（中間成果物）がフォルダに残る設計になっているか？ セッションが切れても翌日再開できる
- [ ] フォルダの役割が明確で、エージェントが指示なく適切な情報源に辿り着けるか？（「議事録なら minutes/」「現状把握なら reports/」等）
- [ ] 新しいスキルを作るとき「入力はどこから取る？」「出力はどこに置く？」がフォルダ構成から自明に決まるか？
- [ ] 操作は1箇所（ターミナル等）から発信し、届け先（Notion、Slack、Gmail等）は複数、という構造になっているか？
- [ ] フォルダ構成は最初から設計するのではなく、散らかってきたら整理する形で育てているか？ ローカルフォルダは後からいくらでも変えられる

### 10-6. スキル設計原則（UNIX哲学の再現）

> 「小さく作って、シンプルに繋ぐ」。
> CLAUDE.mdは文脈（このプロジェクトでの解釈）、スキルは手順（情報をどう処理するか）。両方あって初めて自律的な判断が可能になる。
>
> **スキル定義のテンプレート（SKILL.md）**:
> ```
> スキル名: /skill-name
> 説明:   このスキルが何をするかの1行要約
> 前提:   実行前に満たすべき条件（例: 対象日のページがNotionに作成済みであること）
> 手順:
>   1. 〇〇を △△/ から取得
>   2. □□を ◇◇/ から取得
>   3. ...
>   N. 統合・変換して出力先に書き込む
> ```
> 各手順で入力元（どのフォルダ/サービス）と出力先を明示する。フォルダ構成が決まっていればI/Oは自然に書ける。

- [ ] CLAUDE.md（文脈）とスキル（手順）は分離されているか？ スキルは汎用的な情報処理、CLAUDE.mdがプロジェクト固有の判断基準を担う
- [ ] CLAUDE.mdにはステークホルダーの関係性、ドメイン固有の用語定義、業界構造の制約、頻出する検索クエリのテンプレート等が含まれているか？
- [ ] 各スキルのSKILL.mdは「スキル名・説明・前提条件・手順（I/O明示）」の構造で定義されているか？
- [ ] 各スキルは「1つのことをうまくやる」単機能で設計されているか？（UNIX哲学）
- [ ] 基盤ツール（CLI）→ アトミックなスキル → 複合スキル（パイプライン）のレイヤー構造になっているか？ レイヤーが分かれていれば組み替え・差し替えが容易
- [ ] 複合スキルの入力ソースは明示されているか？（例: /fill-external-minutesは minutes/内部議事録 + reports/状況レポート + data/データ進捗 + Notion検証メモの4ソース統合）
- [ ] スキルのリファクタサイクル（作る → 役割が被る → 整理する）を継続的に回しているか？ リファクタのトリガー: 自分でも「どっち使うんだっけ？」と混乱する、AIに聞いても微妙な回答が返ってくる。そうなったら「このスキルとこのスキル、役割が被ってるけど、データの流れ的にどう整理すればいい？」とAIに壁打ちする。統合されるスキルもあれば、用途が明確に分かれるスキルもある
- [ ] スキルを作るたびに「何をするか」をmdファイルとして言語化させているか？ この蓄積が業務可視化の土台になるだけでなく、コンテキスト資産として突発対応にも複利で効く（10-2参照）

> **再利用可能なスキルパターン（PM業務共通）**:
> スキルの手順自体はプロジェクト横断で再利用できる。プロジェクト固有の判断（何をフィルタし何を残すか等）はCLAUDE.mdが担うため。
>
> | スキルパターン | 入力 | 出力 | 基盤ツール例 |
> |---|---|---|---|
> | 文字起こし→構造化議事録 | minutes/音声ファイル | minutes/.txt + Notion議事録ページ | Whisper API（ptrans等） |
> | 議事録テンプレート作成 | 日付 | Notion議事録DBにテンプレページ | Notion MCP |
> | 議事録→内部チーム共有 | Notion議事録 | Slackチャンネル投稿 | Slack MCP |
> | 複数ソース統合→外部向け変換 | minutes/ + reports/ + data/ + Notion | Notion外部議事録ページ | Notion MCP |
> | 議事録→PDF化→メール送付 | Notion議事録 | PDF + メール下書き | Notion PDF export + Gmail CLI |
> | メール検索 | 検索クエリ | メールスレッド内容 | Gmail CLI（gog-cli等） |
> | CSV→SQL集計 | data/CSVファイル | 集計結果 | DuckDB CLI |
>
> 「複数ソース統合→外部向け変換」は手順は共通だが、変換ルール（何を内部に留め何を外部に出すか）はCLAUDE.mdのドメイン文脈に依存する。

### 10-7. 業務フローの設計（定常/非定常の2レイヤー）

> 業務フローは「週次サイクル（定常）」と「非定常」の2レイヤーで構成される。
> 定常業務を先にパイプライン化し、非定常は個別スキルで対応する。
>
> **定常（週次サイクル）の参照例**:
> MTG録音 → 文字起こし → 内部議事録 → 外部議事録テンプレ作成 → 複数ソース統合・外部向け変換 → 外部MTG → 議事録ブラッシュアップ → PDF化・メール送付
> 使用スキル: /transcribe-to-minutes → /share-internal-minutes → /create-minutes → /fill-external-minutes → /transcribe-and-update → /share-minutes
>
> **非定常の参照例**:
> メール検索・返信ドラフト（/gog-gmail, /email-search）、CSV集計（/duckdb-csv）、レポート更新（/notion-project, /gh-poc-repo）

- [ ] 自分の業務を「定常（毎日/毎週繰り返す）」と「非定常（都度発生）」に区分しているか？
- [ ] 定常業務のパイプライン化を優先しているか？ 繰り返し頻度が高い業務ほどスキル化・パイプライン化の効果が大きい
- [ ] パイプラインは人間ステップ（ドメイン知識を要する情報変換）とAIスキル（定型的な情報変換）の交互配置で設計されているか？ 全自動ではなく、人間の判断ポイントを明示的に組み込む。具体的な交互パターン:
  1. 内部MTG — DSメンバーと評価指標・検証方針を議論 **[人間: ドメイン知識]**
  2. /transcribe-to-minutes — 録音→文字起こし→構造化議事録をNotionに登録 **[AI: 定型変換]**
  3. Claude Codeと壁打ち — 検証方針や技術的論点を構造化・整理 **[人間: ドメイン知識]**
  4. /create-minutes + /fill-external-minutes — 外部テンプレ作成→4ソース統合→内部→外部変換 **[AI: 定型変換]**
  5. 外部MTG — 検証結果をステークホルダーに翻訳、画面共有しながら定例 **[人間: ドメイン知識]**
  6. /transcribe-and-update — 外部MTG録音→文字起こし→議事録ブラッシュアップ **[AI: 定型変換]**
  7. /share-minutes — PDF化→要約→メール下書き作成 **[AI: 定型変換]**
- [ ] AIが定型変換を圧縮することで、人間がドメイン知識変換に集中できる構造になっているか？
- [ ] 非定常業務はアトミックなスキル単体で対応し、無理にパイプライン化していないか？

### 10-8. 業務可視化と再現性

> ボトムアップで積み上げたスキル群・議事録・レポートをAIに全部読ませると、
> 業務フロー全体がリバースエンジニアリングされ、引き継ぎ・自律化の土台になる。

- [ ] スキル群・議事録・レポートが蓄積されたタイミングで、AIに「自分の業務を分析して」と全ファイルを読ませてリバースエンジニアリングさせているか？ やり方: スキル定義、議事録、レポートを全部読ませて「自分の業務を分析して」と頼むだけ。自分でも整理しきれていなかった「毎日何をどの順番でやっているか」が構造化ドキュメントとして出てくる
- [ ] リバースエンジニアリングの結果として、以下が生成されているか？
  - **週次サイクルの俯瞰図** — 月〜金の業務パターン、日次/週次/非定常の3レイヤー構造
  - **情報フロー図** — どの情報がどのスキルを通ってどう変換されるかの全体マップ
  - **個別業務の手順書** — 各業務の入力・出力・使用スキル・判断ポイント（gura氏の例では6本生成）
- [ ] 属人化していたPM業務が、スキル群＋フロー図として再現可能な構造になっているか？ 別のメンバーにスキル群とフロー図をそのまま渡せば同じ業務を再現できる状態か？
- [ ] 自律化の土台（cronや外部トリガーとの組み合わせによるパイプラインの自動実行）は検討されているか？

## 使い方

1. **プロジェクト開始時**: カテゴリ0〜3を重点的に確認し、方針を固める。特に0-7（契約ライフサイクル）と3-5〜3-6（デポジット・再委託先管理）は初期に設計する
2. **設計・開発時**: カテゴリ4〜7の技術的項目をチェックする
3. **リリース前**: 全カテゴリの未チェック項目を洗い出し、リスクとして管理する
4. **運用時**: カテゴリ8を中心に継続的改善のサイクルを回す
5. **常時**: カテゴリ9のエビデンス管理を継続する。危機が起きてからでは遅い
6. **AIエージェント導入時**: カテゴリ10を確認し、SoR/SoAの区分とボトムアップ構築の方針を固める
7. **Skill化時**: 本チェックリストを対話的に質問するエージェントの知識ベースとして使用する

> **プロジェクト規模による適用ガイド**:
> - **小規模（PoC・検証）**: カテゴリ0, 1, 3, 6 を重点的に
> - **中規模（部門導入）**: 全カテゴリをチェック。6-7〜6-10は該当する場合のみ
> - **大規模（全社・本番）**: 全カテゴリ全項目を網羅的にチェック
> - **外部クライアント案件**: カテゴリ0-7, 3-5, 3-6, 9 は規模に関わらず必須
