# 案件知見の資産化ガイド

> 高コンテクストな案件経験（スコープコントロール、契約戦略、クレーム対応等）を
> 再利用可能な資産に変換するためのプロセスガイド。
>
> gura氏のAIエージェント業務適用の知見「手動→繰り返し発見→スキル化」と同じ原則に基づく。
> **1件の案件からの汎用化は早すぎた最適化。まず1次情報を保全し、2件目で共通パターンを抽出する。**

---

## 原則

### やるべきことの区分

| タイミング | やること | やらないこと |
|---|---|---|
| 案件進行中〜直後 | 1次情報の保全（Phase 1） | 汎用フレームワーク化 |
| 案件進行中（都度） | テンプレートへのマージ判断 | — |
| 2件目の類似案件 | パターン抽出・チェックリスト反映（Phase 2） | 全案件共通の仕組み設計 |
| 3件目以降 | スキル化・自動化の検討（Phase 3） | — |

### テンプレートマージとスキル化は別の判断

**テンプレートへのマージは都度判断。** 案件進行中に「これはテンプレートに入れるべき」と判明した知見は、2件目を待たずにマージする。例: K-AI案件で合意管理の原則（3点）が導出された時点でテンプレートに追加した。テンプレートは常に更新されており、2件目の時点では差分が判断できなくなるため。

**スキル化は2-3件の確認が必要。** 自動化・半自動化のためのスキルは、1件の経験では「案件固有の事情」と「共通パターン」の区別がつかない。2件以上で同じ処理を手動で繰り返してから初めてスキル化する。

### なぜスキル化に複数件が必要か

- 1件の案件知見では「この案件固有の事情」と「エンプラ案件共通のパターン」の区別がつかない
- 早期に汎用化すると、特殊ケースを一般則として誤って定着させるリスクがある
- 2件の具体事例を並べて初めて「共通して効いたこと」が特定できる
- これはgura氏の「個別に作ったスキルがたまたま同じだった場合にのみ共通化する」と同じ原則

## Phase 1: 1次情報の保全（案件進行中〜直後）

### 目的
経験の記憶が劣化する前に、生の知見をそのまま書き出す。構造化・抽象化はしない。

### 保存先
`_pm-knowledge/case-studies/{コードネーム}-{年}.md`

### 匿名化ルール（3層構造）

知見はプロジェクト→ケーススタディ→テンプレートの順に流れる。各層で情報の粒度が異なる。

```
プロジェクト層（実名・全詳細）         例: kurashiteku/
  ↓ 知見抽出 + 匿名化
ケーススタディ層（コードネーム・役割名） 例: case-studies/k-ai-2026.md
  ↓ 汎化
テンプレート層（固有名詞なし・原則のみ） 例: _pm-knowledge/（テンプレート本体）
```

| 層 | 会社名 | 人名 | 金額・日付 | 技術詳細 |
|---|---|---|---|---|
| プロジェクト | 実名 | 実名 | そのまま | そのまま |
| ケーススタディ | コードネーム | 役割名 | そのまま（知見の具体性に必要） | そのまま |
| テンプレート | なし | なし | なし（必要なら「N万」等で例示） | 汎化 |

匿名化の手順:
1. プロジェクト開始時にコードネームを決める（例: K-AI）
2. ケーススタディ作成時に、コードネーム＋役割名マッピングを冒頭に記載
3. 本文中の実名を役割名に置換（「前出さん」→「クライアント技術担当」等）
4. テンプレートからケーススタディを参照する際はコードネームで（「ケーススタディK-AI §2参照」）

マッピング例:
```
コードネーム: K-AI
クライアント企業 → K-AI
クライアント意思決定者 → 一岡 → 「意思決定者」
クライアント技術担当 → 前出 → 「技術担当」
受注元PM → 北嶋 → 「受注元PM」
営業担当 → 柳澤 → 「営業担当」
開発パートナーリード → 田村 → 「開発リード」
```

**プロジェクト層のファイルは匿名化しない。** 実務で使うファイルに役割名を使うと誰のことかわからなくなる。匿名化はケーススタディ層への書き出し時に行う。

### 書き出す内容

以下の観点で、**実際に何が起きて、どう判断して、結果どうなったか**を時系列で記録する。

#### スコープコントロール
- スコープクリープが発生した具体的なエピソード
- 「やらないこと」の合意がどう崩れたか/守られたか
- 変更管理が機能した場面、しなかった場面

#### 契約・予算
- 請負/準委任の使い分けで判断に迷ったポイントと結論
- 予算管理で想定と実態がズレたポイント
- 「契約書にこれを入れておくべきだった」と後から気づいた項目
- デポジット精算で問題になったこと

#### 納品物・納期
- 納品物の定義で揉めたポイント
- 納期調整の交渉で効いたこと/効かなかったこと
- 成果物の品質基準がどう合意されたか

#### R&D（準委任）特有の論点
- 探索的な作業のスコープ・成果の定義方法
- 「成果が出なかった」場合の扱い
- 請負部分との境界の引き方

#### クレーム対応
- 何がトリガーで、どう展開して、どう着地したか
- 先方の主張と、こちらの反論の論点整理
- エビデンスが役に立った場面/足りなかった場面
- 弁護士・経営層のエスカレーション判断

#### 判断の根拠と結果
- 重要な意思決定のポイントで、何を根拠にどう判断したか
- その判断は結果的に正しかったか？ 別の選択肢はあったか？

### 書き方のルール

> **最重要原則: 具体知見の再現可能性**
> case-studyに書いた内容から、元の具体的な知見を完全に復元できなければ意味がない。
> 「具体知見 → MDファイル → 再度具体知見」の往復が成立するかを常にチェックする。
> AIは放っておくと見出しレベルの抽象化をする。これは知見の喪失と同義。
> 具体的な状況・判断ロジック・数字・会話の要旨・感情まで含めて記録する。

- **抽象化しない**。「スコープクリープに注意」ではなく「2025年X月にYという要求が来て、Zと判断した」
- **判断ロジックを省略しない**。「総合判断で拒否した」ではなく「納期・難易度・信頼性・リソース・パートナー状況のうち、信頼性（先方内で要件が統一されていない）とリソース（受け入れても完成しないリスク）が決定要因で拒否した」
- **数字を残す**。「コスト増」ではなく「開発費1.5〜2倍、月額ランニング+$150〜500（Pinecone $70〜250 + MongoDB $50〜150 + Redis $30〜100）」
- **対処の具体手順を残す**。「技術的に指摘した」ではなく「1.事実を提示 2.除外した技術的理由を再説明 3.コスト差を数字で示す 4.選択権をクライアントに渡す」
- **時系列を残す**。後からパターンを抽出するとき、順序が重要になる
- **感情・直感も書く**。「この時点で嫌な予感がした」等。暗黙知の手がかりになる
- **結果を必ず書く**。判断だけでなく、その結果どうなったかまで

## Phase 2: パターン抽出（2件目の類似案件）

### 目的
2件の具体事例を並べて、共通して効いたパターンを特定する。

### プロセス
1. 2件目の案件でもPhase 1を実施（1次情報を保全）
2. 2件のcase-studyファイルを並べて読む
3. 共通パターンを抽出: 「両方で同じことが起きた」「両方で同じ対処が効いた」
4. 案件固有の事情を除外: 「片方でしか起きなかった」「特殊な条件に依存していた」
5. 共通パターンをチェックリスト（ai-pm-requirements-checklist）に反映

### 反映先の判断基準
- **チェック項目として汎用的か？** → チェックリストに追加
- **特定の状況での対処法か？** → case-studyに残し、チェックリストからはリンクで参照

---

## Phase 3: スキル化（3件目以降）

### 目的
繰り返し確認されたパターンをClaude Codeのスキルとして構造化し、自動化・半自動化する。

### 検討対象の例
- 契約レビューチェックリストの自動適用
- リスクアセスメントの定期実行
- エビデンス保全状況の棚卸し
- 案件知見の構造化テンプレート生成

### 判断基準
- 3件以上で同じパターンが確認されている
- 手動で繰り返すコストが明確にある
- スキル化しても判断品質が落ちない（定型変換に該当する）
