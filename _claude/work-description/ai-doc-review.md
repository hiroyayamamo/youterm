# タスク定義書: AI生成ドキュメントのレビュー

## 概要
クライアントがAIで生成したドキュメント（要件書、仕様書等）を受領した際に、既存合意との矛盾を検出し、対応方針を策定する。

## トリガー
- クライアントからAI生成ドキュメント（PDF、テキスト）を受領した時
- 技術要件書・仕様書のバージョン更新を受領した時

## 入力
- 受領したドキュメント
- logs/meeting_decisions.md（合意事項）
- logs/scope_log.md（スコープ）
- logs/tech_decisions_log.md（技術選定）
- logs/budget_log.md（予算）

## 手順
1. 受領ドキュメントをminutes/に保存（ルーティングルール: 一次着地点）
2. /check-ai-doc スキルで矛盾チェックを実行
3. 矛盾が検出された場合:
   a. 矛盾レポートを作成（矛盾箇所/既存ソース/信頼度/判定）
   b. 矛盾の種類を分類:
      - 配置バグ（修正漏れ）→ 先方に修正依頼
      - 除外済み項目の復活 → スコープ変更として変更管理プロセスに載せる
      - 新規概念の追加 → 合意の有無を確認
      - 先方内部矛盾の反映 → 先方に内部統一を依頼
   c. 対応方針を策定: 技術的理由を数字付きで説明 + コスト影響 + 選択肢の提示
4. 矛盾がなかった場合: その旨を記録し、合意済みとして扱う
5. 結果をlogs/に追記（/add-log スキル経由）

## 判断ポイント
- **「受け入れて増額」vs「拒否して元に戻す」**: 納期・難易度・信頼性・リソース・パートナー状況の5要素で総合判断。詳細→refs/pm_playbook.md「AI生成ドキュメント対応ルール」
- **ワードスロップの検出**: クライアントが技術用語を使っているが内容を理解していない兆候がある場合、メタ的な問題提起を行う
- **ループの兆候**: 同じ指摘を2回以上繰り返す場合、「コンテキスト固定」または「要件定義の巻き取り」を提案

## 出力
- minutes/にドキュメント保存
- 矛盾レポート（会話中に報告）
- logs/への追記（該当するログファイルに振り分け）

## 対応スキル
- /check-ai-doc（矛盾チェック自動化）
- /add-log（ログ追記）
