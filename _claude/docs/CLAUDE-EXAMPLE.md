# SmartOrder（飲食チェーン向けAI注文予測システム）

## Overview
ゼンショクホールディングス（全国280店舗の飲食チェーン）向けに、過去の売上・天候・イベント情報から日次の食材発注量をAI予測するシステムを開発する。Phase1（PoC: 10店舗）で予測精度を検証し、Phase2以降で全店舗展開する。

## ドメイン知識
### 用語定義
| 用語 | 意味 | 補足 |

| 日販 | 1日あたりの売上高 | 文字起こしで「にっぱん」→「日本」に誤変換されやすい |
| フードロス率 | 廃棄食材量 / 仕入食材量 | 現状平均4.2%。目標3.0%以下。KPIとして先方経営会議で報告される数字 |
| PAR値 | Periodic Automatic Replenishment。品目ごとの適正在庫基準量 | 現在はExcelで店長が手動設定。これをAI化するのがPJの目的 |

| 発注リードタイム | 発注から納品までの時間 | 生鮮: 翌日、冷凍: 3日、乾物: 1週間。品目カテゴリで異なる |

### 業界構造・制約
- 食品衛生法: 食材の温度管理・消費期限管理が法的義務。予測が外れて欠品→代替食材使用時もアレルゲン表示義務あり
- 飲食チェーンの発注サイクル: 生鮮は毎日15時締切で翌朝納品。予測システムは14時までに推奨発注量を提示する必要がある
- POS データの制約: ゼンショクHDのPOSは2019年導入のNECシステム。APIなし、CSVエクスポートのみ（日次バッチ、前日分が翌朝9時に生成）
- 個人情報: POSデータに顧客個人情報は含まないが、従業員シフトデータはマイナンバー紐付けのため連携不可。シフトは「人数×時間帯」のみ受領可
- 競合状況: 先方はRetailNext社のソリューションも比較検討中（山本部長が2026/3の展示会で接触）。価格優位性を保つ必要あり

## 現状サマリー（2026/04/13時点）
Phase1（PoC）は10店舗中6店舗でデータ取得・モデル学習が完了。残り4店舗は POS CSV のフォーマット不整合（2019年以前の旧フォーマット混在）で田中が修正対応中、4/18完了見込み。

予測精度はA品目でMAPE 12%（目標15%以下）を達成しているが、天候急変時（降雪・台風）の精度が30%超に悪化する課題あり。田中が天候特徴量の追加を検証中。

山本部長から4/10のMTGで「Phase2の予算取りのために5月の経営会議に中間報告を出したい」と要望あり。4/25までにPoC中間レポートを納品する必要がある（logs/scope_log.md 2026-04-10エントリ参照）。

契約面: Phase1の準委任契約（月額250万 x 6ヶ月 = 1,500万）は2026/2/1開始、7/31終了。Phase2への移行判断は6月末。請負移行の場合の見積もりを5月中に提示する約束（logs/contract_log.md 2026-03-15エントリ参照）。

リスク: 高橋（情シス）からPOSデータのクラウド連携にセキュリティ審査が必要と4/8に連絡あり。審査に最大2ヶ月かかる可能性。Phase2のアーキテクチャ（クラウドAPI化）に影響するため、審査プロセスの前倒しを山本部長経由で依頼中。

## 鉄則
1. SmartOrder固有のルール: 店舗データ（売上、在庫、シフト）は匿名化してからログに記載する。店舗名はコードネーム（A店、B店）を使う。日販の実数値は記載可だが、店舗と紐付けて特定できる形にしない（例: 「A店の日販85万」はNG、「対象店舗の日販レンジ: 60-120万」はOK）
2. スキルを作るたびに業務の背景をmdに言語化させる（コンテキスト資産の蓄積）
3. 2-3回同じことを手動で頼んだらスキル化する
4. 内部向け/外部向けの情報境界を常に意識する
5. 合意事項はlogs/に追記型で保全する。散逸させない
6. ファイルを追加したらインデックスも更新する
7. 新しい情報（文書・Slack・議事録・資料等）を受け取ったら、内容への対応の前にルーティングルールを確認し、格納先を決定・保存する。保存してから分析・回答する
8. ログエントリには必ず日付/ソース/信頼度を付ける
9. 検収条件は「機能の有無」で判定する。品質指標（精度%、使いやすさ等）を検収条件に入れると永遠に検収が通らない。詳細→refs/pm_playbook.md
10. AI生成ドキュメントは受領したらまず矛盾チェック。詳細→refs/pm_playbook.md
11. 進捗管理 = リスク管理。「何%できたか」ではなく「何がまだわかっていないか」を追う。コード行数・チケット消化率・テスト通過率は進捗指標として機能しない
12. 記録は文字、理解は口と絵。文字は合意の証跡として残す。相手に理解させるフェーズは口頭+画面共有。文字で理解させようとしない
13. 構造変更（リストラクチャリング・テンプレート化・バックアップ復帰・ファイル移動統合分割）の後はdiffで内容ロスがゼロであることを検証する。手順→WRITING_RULES.md「構造変更時の検証ルール」
14. 契約は先方が前向きなうちに金額・スコープを書面で固める。雰囲気が良い時にリスクは表面化しない。中盤以降の悪化に常に備える。詳細→refs/pm_playbook.md「契約タイミングの原則」
15. 先方が不要と言ってもフェーズごとの定型ドキュメントは最低限作成・共有する。省略の合意は書面で記録する。詳細→refs/pm-document-guideline.md
16. 仕様書は実装前に4観点並列レビュー（セキュリティ、スケーラビリティ、エラーケース、実装者シミュレーション）をかける。詳細→refs/pm_playbook.md「仕様書の実装前レビュー」
