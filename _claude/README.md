# PMプロジェクトテンプレート

Claude Code + Markdown で動くプロジェクト管理基盤。`cp -r` でコピーして使い始める。

## このテンプレートでできること

### 情報を失わない仕組み（SoR層）

| 機能 | 仕組み | 使うもの |
|---|---|---|
| **合意・決定の追跡** | MTGでの合意を6種のログに振り分け、日付/ソース/信頼度を必ず付与 | logs/ 6ファイル + `/add-log` スキル |
| **信頼度による合意管理** | 「確定合意 > 口頭合意 > こちらの認識 > 先方主張 > 推測」の5段階で情報の確度を可視化 | WRITING_RULES.md |
| **契約防衛の証跡保全** | 書面合意・変更起票・検収を時系列で記録。口頭合意は「書面化待ち」ステータスで管理 | logs/contract_log.md |
| **揮発性ソースの即時保全** | Slack・先方Notion等の相手が改変できるソースを受領時に原本保存 | data/ + ルーティングルール |
| **AI文書の矛盾検出** | AI生成ドキュメントを受領→既存ログと自動照合→矛盾箇所を一覧化 | `/check-ai-doc` スキル |

### 情報を届ける仕組み（SoA層）

| 機能 | 仕組み | 使うもの |
|---|---|---|
| **議事録→ログ自動振り分け** | MTGメモから決定事項を抽出し、ルーティングルールに従って各ログに追記 | `/extract-decisions` スキル |
| **週次レポート自動生成** | logs/ 6ファイルから今週分を集約してレポート化。内部向け/外部向け切替対応 | `/weekly-report` スキル |
| **構造変更の安全検証** | ファイルの移動・統合・分割後に内容ロスがゼロであることをdiffで検証 | `/verify-structure` スキル |
| **インデックス自動保守** | ファイル追加・移動後にREADME.mdインデックスを自動更新 | `/update-index` スキル |

### 立ち上げの流れ

```
Phase 0: cp -r → CLAUDE.mdのプレースホルダーを埋める（記入例: docs/CLAUDE-EXAMPLE.md）
Phase 1: 合意管理の原則を先方と合意 → 契約チェックリストで漏れ確認
Phase 2: 情報蓄積開始 → 2-3回同じ作業を頼んだらスキル化
Phase 3: 蓄積された情報からdocs/に業務可視化を生成
```

## 設計思想

> SoR（情報の記録・保全）とSoA（情報の加工・変換・配信）の両方を1つの構成で実現する。
> - logs/ + WRITING_RULES.md = SoR層（何が決まったか・何が起きたかを正確に残す）
> - minutes/ → reports/ + .claude/skills/ = SoA層（情報を加工して届けるパイプラインを育てる）
>
> 実案件での実践 + gura氏のAIエージェント業務適用の知見を統合。
> **注意**: 案件を重ねるごとに共通パターンを抽出し改善する（knowledge-assetization-guide.md 参照）。

## フォルダ構成

```
{project-name}/
├── CLAUDE.md                    — プロジェクトの脳（マスターインデックス + ルーティング + ドメイン知識）
├── WRITING_RULES.md             — 記録品質ルール（具体知見の再現可能性）
├── collection_routes.md         — 情報収集ルート定義
├── .claude/skills/              — PM業務スキル定義群
│   └── SKILL-TEMPLATE.md
├── logs/                        — 追記型ログ（意思決定・合意の履歴保全）【SoR層】
│   ├── stakeholders.md
│   ├── scope_log.md
│   ├── budget_log.md
│   ├── tech_decisions_log.md
│   ├── meeting_decisions.md
│   └── contract_log.md          — 書面合意・変更起票・検収の履歴
├── minutes/                     — 情報の一次着地点（音声→文字起こし→議事録）【SoA層の入口】
├── reports/                     — 加工済みアウトプット（スナップショット型 + 更新型）
├── docs/                        — 業務棚卸し・リバースエンジニアリング結果
├── work-description/            — タスク定義書（1タスク1ファイル）
├── data/                        — 生データ + エビデンス原本（相手が改変できる文書のPDF等）
└── refs/                        — 参照ファイル（ログからリンクされる詳細資料）
```

## 各ファイル・フォルダの役割

### CLAUDE.md（マスターインデックス + ルーティング + ドメイン知識）
プロジェクトの全文脈を集約。新しいセッションでまず読まれるファイル。
- プロジェクト概要
- ステークホルダー（名前・役割・連絡手段）
- ドメイン知識（用語定義、業界構造・制約）— 文字起こし誤変換の自動修正にも使用
- 頻出検索クエリテンプレート（Gmail、Notion等）
- 報告フォーマット（内部向け/外部向け/変換ルール）
- ルーティングルール: 新情報 → どのファイル/フォルダに格納するか
- Key Files インデックス: 全ファイルの一覧と役割・内容キーワード
- 情報管理の原則（スナップショット型 vs 更新型 vs 追記型）
- インデックスによるコンテクスト量の制御方針
- 鉄則（プロジェクト固有ルール + 共通ルール）
- 現状サマリー

### WRITING_RULES.md（記録品質ルール）
ログ・議事録・参照ファイルの記録品質を担保するルール。
- 具体知見の再現可能性の原則（セルフチェック・アンチパターン）
- エントリフォーマット定義（日付 / ソース / 信頼度 / 内容）
- ソース分類の定義
- 信頼度分類: 確定合意 / 口頭合意 / こちらの認識 / 先方主張 / 推測
- ログと参照ファイルの使い分け基準（概要→ログ、詳細→refs/にリンク）
- 数字の記録ルール（「コスト増」ではなく具体金額）

### collection_routes.md（情報収集ルート）
情報の取りこぼしを防ぐための収集ルート定義。
- ソース × 収集方法 × 担当 × 頻度 × 格納先のテーブル
- 収集時の注意事項
- 情報受領時のフロー

### .claude/skills/（スキル定義群）
PM業務のビルディングブロック。ボトムアップで育てる。
- SKILL-TEMPLATE.md: スキル名・説明・前提・手順（I/O明示）のテンプレート
- 2-3回同じことを手動で頼んだらスキル化する
- スキル作成時に業務の背景をmdに言語化させる（コンテキスト資産の蓄積）

### logs/（追記型ログ）【SoR層】
意思決定・合意の履歴を追記型で保全する。Context Rot防止の核。

| ファイル | 記録内容 |
|---------|---------|
| `stakeholders.md` | ステークホルダーの役割・権限・懸念・関係変遷 |
| `scope_log.md` | スコープ・機能合意の変更履歴 |
| `budget_log.md` | 予算・見積もり・精算の推移 |
| `tech_decisions_log.md` | 技術選定・アーキテクチャの決定履歴 |
| `meeting_decisions.md` | MTGでの決定事項・重要発言 |
| `contract_log.md` | 書面合意・変更起票・検収判定の履歴 |

各エントリに必須: **日付 / ソース / 信頼度 / 内容**
詳細は refs/ に置き、ログからリンクする（「詳細 → refs/filename.md」形式）

#### contract_log.md の設計意図
他の5本のログは「何が起きたか」を記録するが、contract_log.mdは「何に合意したか・何が変更されたか」を法的根拠として記録する。
記録対象:
- 書面合意（覚書・発注書・見積もり承認）の締結日・内容・署名者
- 変更起票（変更要求の受領・影響評価・追加見積もり・合意/却下）
- 検収判定（納品・確認期間・検収完了/みなし検収適用）
- 口頭合意の書面化状態（口頭のみ / 書面化済み / 書面化待ち）

このログがあれば、後から「そのつもりではなかった」と言われた時に「この日に書面で合意しています」と返せる。

### minutes/（情報の一次着地点）【SoA層の入口】
外部から入ってくる情報の最初の着地点。同じフォルダ内で段階的に加工される。
- 音声ファイル → 文字起こしテキスト → 構造化議事録
- README.mdにインデックス（日付・種別・要点）を維持
- minutes/の議事録から意思決定・合意事項を抽出 → logs/に追記（ルーティング）

### reports/（加工済みアウトプット）
minutes/の一次情報を分析・統合して、届けるための形にしたもの。
- **スナップショット型**（日付付き新規ファイル）: 週次報告、状況更新、ステークホルダー向け資料
- **更新型**（1ファイルを更新）: 合意事項トラッカー、現行スコープ定義、体制表
- 変更履歴はgitで追跡

### docs/（業務棚卸し・可視化結果）
スキル群・議事録・レポートが蓄積されたタイミングで、AIに全ファイルを読ませてリバースエンジニアリングさせた結果。
- 週次サイクル俯瞰図、情報フロー図、個別業務手順書
- 最初から作らない。十分蓄積されてから生成する

### work-description/（タスク定義書）
各業務タスクの手順と目的を1タスク1ファイルで明文化。
- スキルと対応するタスク定義を作る
- 引き継ぎ・オンボーディングにも使える

### data/（生データ + エビデンス原本）
CSV等の分析用生データに加え、**ソースの管理権が相手にある文書の原本**を保管する。
加工せずそのまま保管（加工結果はreports/に置く）。

保管対象の判断基準: **相手が改変・削除できるか**
- 保管する: 先方のGoogle Sheets/Notion/Slackから受領したPDF・スクリーンショット、見積書、発注書、覚書、AI生成要件書の各バージョン
- 保管不要: 自分のGitリポジトリ内のmd、自分が管理するTLDV録画、公開情報

運用:
- README.mdにインデックス（ファイル名・受領日・ソース・用途）を維持
- logs/のエントリから「原本 → data/filename.pdf」形式でリンクする
- 機密データ・個人情報は置かない

### refs/（参照ファイル）
ログからリンクされる詳細資料の格納場所。
- 種類: 分析・戦略 / 計画・方法論 / 防御プレイブック / 対外出力 / R&D研究

## 情報の流れ

```
[外部情報]
    │
    ▼
minutes/（一次着地点: 音声→文字起こし→議事録）
    │
    ├──→ logs/（意思決定・合意事項を抽出して追記）【SoR: 記録・保全】
    │
    ├──→ reports/（分析・統合して届ける形に加工）【SoA: 変換・配信】
    │        │
    │        └──→ [Notion/Slack/Gmail/PDF]（ステークホルダーに配信）
    │
    └──→ data/（定量データがあれば格納）
```

minutes/が入口、logs/とreports/が出口。
logs/は「何が決まったか」を残す（SoR）、reports/は「届ける形に加工する」（SoA）。
この二股のルーティングがCLAUDE.mdに定義される。

### MCP サーバー（ユーザーレベル設定）

以下のMCPサーバーはプロジェクト外のユーザーレベル設定（`--scope user`）で追加する。全プロジェクト共通で使用できる。

```bash
# Draw.io — ER図・フロー図等の図表生成
claude mcp add-json drawio '{"type":"stdio","command":"npx","args":["-y","drawio-mcp-server","--editor"]}' --scope user
```

設定後、Claude Codeセッションを再起動すると有効になる。

グローバル側が更新された場合、WRITING_RULES.md も確認・同期が必要。

## 立ち上げ手順

### Phase 0: テンプレートのコピーとPJ固有情報の記入
1. このテンプレート（`_pm-knowledge/`）をプロジェクトディレクトリにコピーする（`cp -r _pm-knowledge/ {project-name}/`）
2. CLAUDE.md のプレースホルダーを埋める: プロジェクト名、Overview、ステークホルダー、ドメイン知識（用語定義・業界制約）、頻出検索クエリ、報告フォーマット（これらの情報はチャット形式でCLIに直接打ち込んでもよい）
3. collection_routes.md を情報源に合わせて調整（ソース×収集方法×担当×頻度）Slackから取得、メールから取得などプロジェクトの実態に合わせる
4. WRITING_RULES.md のソース・信頼度分類をプロジェクトに合わせて調整（必要な場合のみ）

### Phase 1: 合意の土台を固める
5. **合意管理の原則3点を合意する**（refs/pm_playbook.md参照。伴走型でも請負型でも必要）
6. **請負型の場合は書面合意を設計する**（refs/pm_playbook.md「変更管理プロセス」「書面合意に含める項目」参照。開発着手前に完了させる）
7. refs/ai-pm-requirements-checklist-V3.md を使って、プロジェクト固有のリスク・論点を洗い出す

### Phase 2: 情報の蓄積開始
8. 情報が入り次第、CLAUDE.mdのルーティングルールに従ってlogs/に追記開始
9. 業務を手動でClaude Codeに頼み始める（スキル化はまだしない）
10. 2-3回同じことを手動で頼んだらスキル化。.claude/skills/ にSKILL.mdを作成

### Phase 3: 可視化と改善
11. スキル群・議事録・レポートが蓄積されたらdocs/にリバースエンジニアリング結果を生成
12. 案件知見をcase-studies/に書き出す（refs/knowledge-assetization-guide.md参照）

## カスタマイズポイント

- **ログファイルの種類**: 6本が基本だが、プロジェクトの性質により増減（例: 法務ログ、品質ログ）。contract_log.mdは請負案件では必須
- **ソース分類**: プロジェクトの情報源に合わせて追加・変更
- **信頼度分類**: 基本の5段階は汎用的だが、組織文化に合わせて調整
- **収集ルート**: プロジェクトのツール・体制に合わせて定義
- **ルーティングルール**: ログ・フォルダの種類に連動して更新
- **フォルダ構成**: gura氏の7フォルダが基本だが、logs/とrefs/を加えた9フォルダがフル構成。不要なフォルダは使わなくてよい
- **変更管理プロセス**: CLAUDE.mdテンプレートに項目リストを配置。プロジェクトに合わせて合意項目を追加・削除
- **検収条件**: MVPでは「機能の有無」が基本。品質指標を入れるかはフェーズとクライアントの特性で判断
